Les programmes qu'on vous fournit : réalité et solutions

⏱ 12 min de lecture🏷 Programmes · Incompatibilités · Diagnostic · Méthode

Dans la réalité d'atelier, un technicien d'usinage reçoit régulièrement des programmes CN de sources externes — du client, d'un prestataire, d'un ancien opérateur. La situation la plus courante : ces programmes sont supposés fonctionner sur votre machine, et dans la pratique ils ne peuvent pas être lancés tels quels. Voici une situation réelle, anonymisée, et la méthode pour y faire face.

Le contexte typique

Vous recevez un lot de programmes pour une pièce à usiner. Sur le papier, tout est là — OP1, OP2, programmes d'outillage. En réalité, aucun ne peut être lancé directement. Voici les 11 catégories de problèmes que vous allez rencontrer :

Problème 1 : Révision de plan incorrecte

Les programmes sont écrits pour une révision du plan qui n'est plus la révision en vigueur. Entre la révision A et la révision B, des cotes ont changé, des opérations ont été ajoutées ou supprimées. Sans vérification systématique, vous usineriez des pièces non conformes au plan actuel.

⚠️ Toujours commencer par comparer la révision du plan dans les programmes reçus avec la révision du plan de fabrication actuel. Si elles diffèrent, les programmes sont à analyser cote par cote.

Ce qu'il faut vérifier en priorité :

  • Les cotes tolérancées critiques (H7, G6, H6) — leur valeur peut avoir changé
  • Les rayons dans les angles — un rayon peut passer de R12 à R3 entre deux révisions
  • Les opérations présentes / absentes — une révision peut ajouter un alésage ou supprimer une rainure

Problème 2 : M-fonctions machine-spécifiques

Les M-fonctions au-delà de M30 sont définies par le constructeur machine et varient d'une machine à l'autre. Un programme écrit pour une machine X peut contenir des L M61 ou L M62 qui correspondent à des fonctions inexistantes sur votre machine.

Sur TNC 426, si une M-fonction inconnue est rencontrée, le comportement varie selon le paramétrage machine — arrêt immédiat, erreur, ou exécution sans effet. Dans tous les cas : imprévisible.

Solution : identifier toutes les M-fonctions non standards (M35 et au-delà) dans le programme, consulter le manuel de votre machine pour vérifier leur existence, supprimer ou remplacer celles qui sont inconnues.

Problème 3 : Syntaxe incompatible avec votre version de TNC

Certaines instructions existent sur les TNC modernes (iTNC 530, TNC 640) mais pas sur TNC 426 :

  • F AUTO — avance calculée automatiquement. Syntaxe iTNC 530+ uniquement. Erreur de syntaxe sur TNC 426.
  • Q206= AUTO, Q207= AUTO, Q209= AUTO — même problème dans CYCL 230.
  • CYCL DEF 7.1 #N — sélection de point zéro par numéro avec syntaxe # — peut être incompatible selon la version firmware.

Solution : rechercher systématiquement AUTO et # dans les programmes reçus. Remplacer chaque F AUTO par la valeur d'avance correspondant à l'outil. Remplacer les CYCL DEF 7.1 #N par CYCL DEF 247.

Problème 4 : TOOL CALL sans numéro d'outil

Un bug de programmation classique : TOOL CALL Z S1500 F80 sans numéro d'outil. Aucune machine ne peut interpréter ça correctement — soit erreur, soit chargement d'un outil aléatoire.

Ce type d'erreur passe inaperçu à la relecture rapide. Il faut chercher méthodiquement tous les TOOL CALL dans le programme et vérifier que chacun a un numéro d'outil.

; ❌ Bug : TOOL CALL sans numéro
TOOL CALL  Z S1500 F80

; ✅ Correct :
TOOL CALL 14 Z S1500 F80

Problème 5 : Noms de sous-programmes incorrects

Un CALL PGM appelle un sous-programme par son nom de fichier. Si le fichier fourni s'appelle SOUS-PROG-OP1.H et que le programme principal fait CALL PGM SOUS-PROG.H, la machine s'arrête avec "programme introuvable".

Vérifier également que les extensions sont complètes (.H obligatoire sur TNC 426 — certains programmes omettent l'extension).

Problème 6 : Sous-programmes FAO illisibles

Les programmes d'ébauche générés par FAO (surfaçage, évidement) sont souvent fournis comme sous-programmes séparés — des milliers de lignes de trajectoires calculées par logiciel, sans en-tête, sans TOOL CALL, sans gestion d'origine.

Ces fichiers posent deux problèmes :

  • Ils supposent que la machine est déjà dans le bon référentiel quand ils sont appelés
  • Ils sont souvent en coordonnées machine de la machine d'origine — incompatibles avec votre système d'origine

Solution : ne pas utiliser ces sous-programmes FAO. Reconstruire les opérations d'ébauche en programmation native — c'est souvent plus rapide à programmer et infiniment plus rapide à l'usinage car les trajectoires natives sont optimisées sans à-vide inutile.

Problème 7 : Erreurs géométriques dans les contours

Des erreurs directement dans les cotes du contour programmé — une face à Y-100 au lieu de Y-83, par exemple. Ces erreurs passent au chargement du programme mais produisent une pièce hors cote ou une collision.

Solution : comparer systématiquement les cotes de contour du programme avec le plan avant le premier lancement. En particulier les cotes globales (largeur, hauteur totale) et les cotes tolérancées.

Problème 8 : Instructions de desserrage manuel en cours d'usinage

Des STOP dans le programme avec instructions de desserrer et resserrer la pièce manuellement entre deux opérations. C'est parfois nécessaire techniquement, mais dans la pratique :

  • Toute reprise de bridage introduit un risque de décalage
  • Le "serrage gentil pour ne pas écraser la pièce" est une consigne subjective — non reproductible d'un opérateur à l'autre
  • Incompatible avec une production série reproductible

Problème 9 : Table outils différente

Les numéros d'outils dans les programmes reçus correspondent à la table outils de la machine d'origine. Si T14 chez eux est une fraise D10 et que T14 chez vous est un foret D3, vous avez un problème majeur.

Solution : identifier tous les TOOL CALL, construire la table de correspondance (leur numéro → leur outil → votre numéro), remplacer systématiquement avant tout essai.

Problème 10 : Origine différente

Les programmes reçus utilisent souvent une origine au centre de la pièce. Si vous travaillez avec une origine sur le bord, toutes les coordonnées sont à convertir. Ce n'est pas une erreur — c'est une incompatibilité de convention.

Problème 11 : Opérations manquantes

Le programme reçu ne contient pas toutes les opérations. Des taraudages absents, des chanfreins oubliés, une opération prévue "au fil EDM" fournie comme programme de reprise en fraisage avec serrage intermédiaire manuel — non applicable en production série.

Méthode de diagnostic avant premier lancement

Face à un lot de programmes inconnus, voici la séquence à suivre avant de mettre quoi que ce soit en machine :

  1. Vérifier la révision du plan dans chaque programme vs le plan actuel
  2. Rechercher les M-fonctions non standards et les identifier
  3. Rechercher "AUTO" — chaque occurrence est une incompatibilité potentielle
  4. Vérifier tous les TOOL CALL — chacun doit avoir un numéro
  5. Vérifier tous les CALL PGM — chaque fichier appelé doit exister avec le bon nom et la bonne extension
  6. Comparer les cotes de contour avec le plan — toutes les cotes globales et tolérancées
  7. Construire la table de correspondance des outils
  8. Vérifier l'origine et la convertir si nécessaire
  9. Identifier les opérations manquantes par comparaison avec le plan
✅ Ce diagnostic complet prend du temps mais évite de mettre une pièce au rebut ou d'endommager la machine. Il est systématique, pas optionnel.

La vraie solution : reconstruire depuis zéro

Après ce diagnostic, la question se pose souvent : vaut-il mieux corriger les programmes reçus ou repartir d'une feuille blanche ?

La réponse dépend du volume de corrections. Si le programme a 3 ou 4 problèmes localisés, la correction est rapide. Si les incompatibilités sont nombreuses, si la révision est incorrecte, si les origines sont différentes et si des opérations manquent — reconstruire depuis le plan est souvent plus rapide et plus fiable. Un programme reconstruit sur la bonne révision, dans le bon référentiel, avec les bonnes conventions, sera meilleur que le programme original corrigé à la hâte.

La reconstruction complète permet aussi d'optimiser l'usinage — organisation par outil, circuit continu sans à-vide, documentation complète — ce qu'un programme corrigé ne permettra jamais d'atteindre.
